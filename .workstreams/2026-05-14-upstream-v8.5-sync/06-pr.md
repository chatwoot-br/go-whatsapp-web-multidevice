# PR — Pull Request

QRSPI PR stage. One PR per slice — or a single bundled PR with per-slice commits if the slices are small.

> Plan review is now a cheap spot-check; the deep review happens against code.

## Per-slice PR template

```
## Slice N — <title>

### What

<one-line summary>

### Why

<links to D + S + P sections>

### How

- <bullet of approach>
- <test plan>

### Risk

- <what breaks if this is wrong>
- <rollback procedure>

### Checks

- [ ] Tests pass
- [ ] Manual round-trip on paired phone
- [ ] CHANGELOG updated
- [ ] No new env vars without docs
```

## Cross-cutting checks (final PR before merging the whole upgrade)

- [ ] Webhook contract document updated (see `wiki/concepts/hmac-signed-webhooks.md`).
- [ ] CHANGELOG.md `[v8.5.0+1]` section complete with Upstream Changes / Fork Changes.
- [ ] chatwoot-app side cutover plan documented (if Q2 = A).
- [ ] Rollback tag `pre-upgrade-snapshot-2026-05-14` still exists.
- [ ] Helm chart version bumped + image tag pin updated.

## Open questions for PR review (not blockers)

- Should fork start tracking upstream more aggressively (monthly sync cadence)?
- Worth opening a PR upstream with BR phone rules so the divergence shrinks long-term?
- Whatsmeow protocol updates suggest a CI gate that auto-flags when fork falls >N commits behind?
